project: party-lottery
cycle: 3
goal: 役割割り当て・結果表示品質・主催者/参加者UXを仕上げ、MVPのMust要件を一通り通す

## Scope（Must）
1. 役割割り当ての実装完了
- 主催者が役割一覧を設定できる
- 抽選時に役割を当選者へ順に割り当てられる
- 順位あり／なしの両方で結果表示できる
- 役割数と当選人数が一致しない場合の扱いをUI/ロジックで統一する

2. 結果表示の完成度向上
- 主催者画面では結果を見やすく表示する
- 参加者画面でも公開ON時に結果が見やすく表示される
- 自分が当選している場合は参加者画面で強調表示する
- 非公開時は誤って結果が見えないようにする

3. くじ引き体験の最低限の演出
- 抽選開始後に即結果表示ではなく、短い演出を入れる
- 順位ありの場合は順次表示、なしの場合はまとめ表示など、ルールに沿った見せ方にする
- リロード後でも結果再表示が壊れない

4. 主催者UXの仕上げ
- ルーム状態、設定内容、結果公開設定が主催者に分かる
- 抽選済みルームでは再実行できないことが明確
- draw画面と主催者トップの導線が迷わない

5. 詳細設計の最適化を含む整備
- 役割割り当ての仕様を固定する
- 結果表示用のViewModelを導入し、APIの生データ依存を減らす
- 公開/非公開・順位あり/なし・役割あり/なしの表示分岐を整理する

## Scope（Should）
- QRコード表示を主催者画面に追加する
- 抽選結果カードの見た目を整える
- 参加者待機画面で状態文言をより分かりやすくする
- hostToken / participantToken がないときの案内を改善する

## Out of Scope
- 再抽選
- 複数抽選パターン管理
- 履歴画面
- 72時間削除Cron
- 高度なアニメーション
- 通知機能
- ログイン機能
- CI / 自動テストの本格導入
- 本番運用向け監視強化

## Detailed Design Notes
1. 役割割り当て仕様
- roles はカンマ区切り入力を配列化して保存する
- 空文字は除外し、trim して扱う
- 抽選時、当選者配列の先頭から roles を順に付与する
- roles が winner_count より少ない場合、残りの role は `null`
- roles が winner_count より多い場合、超過 roles は無視する
- ranked=false でも role は付与可能

2. 結果表示ViewModel
- APIの `results` JSON をそのまま描画せず、画面用の表示モデルへ変換する
- 表示モデルで以下を吸収する
  - rank の有無
  - role の有無
  - 自分かどうか
  - 強調表示フラグ
- 主催者画面と参加者画面で同じ整形関数を使えるようにする

3. 演出方針
- 実装優先で、タイマー制御ベースの簡易演出にする
- ranked=true のときは 1件ずつ reveal
- ranked=false のときは短い待機後に一括表示
- 演出中も最終結果はDBを正とし、再読込時は即結果表示でよい

4. UX方針
- 主催者は「設定済み」「締切済み」「抽選済み」が一目で分かることを優先
- 参加者は「待機中 / 締切済み / 結果公開 / 結果非公開」の4状態が分かることを優先
- 複雑な画面分岐より、状態に応じた明確なメッセージを出す

## Definition of Done
- 役割入力が保存され、抽選結果に反映される
- 順位あり／なしの両方で結果表示が成立する
- 参加者画面で自分の結果が強調表示される
- 結果非公開時に参加者へ結果が表示されない
- 主催者画面で抽選済み状態と結果が安定表示される
- 簡易演出が入る
- リロード後も結果表示が壊れない
- 少なくとも `build` が成功する
