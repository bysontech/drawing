project: party-lottery
cycle: 2
goal: 参加締め切りから抽選実行・結果表示までを縦に通し、主催者が1回の抽選を完了できる状態にする

## Scope（Must）
1. 参加締め切り導線
- 主催者が参加締め切りボタンを押せる
- 締め切り後は新規参加を受け付けない
- 主催者画面に締め切り済み状態が分かる
- 抽選開始は締め切り後のみ可能

2. 抽選設定の最小実装
- 主催者が以下を設定できる
  - 順位あり／なし
  - 当選人数
  - 参加者に結果を見せるか
- 役割割り当ては cycle 2 では「型と保存先まで」を優先し、UIは任意または簡易でよい
- 設定は抽選前に保存できる

3. 抽選実行の最小導線
- 主催者が抽選開始ボタンを押せる
- APIで抽選を1回実行し、結果を保存する
- ルーム状態が `closed` から `drawn` に遷移する
- ルームごとに1回だけ抽選できる

4. 結果表示
- 主催者画面には必ず結果が表示される
- 参加者画面は、結果公開ONのときのみ結果表示に切り替わる
- 結果公開OFFのときは待機画面から案内文を表示する

5. 詳細設計の最適化を含む基盤整備
- 抽選アルゴリズムを先に固定する
- 状態遷移（waiting → closed → drawn）を厳密にする
- `lottery_results` テーブルと結果JSONの責務を明確にする
- 主催者専用結果取得と参加者向け結果取得の扱いを分ける

## Scope（Should）
- 主催者くじ引き画面 `/room/:roomId/draw` を用意する
- 抽選中の簡易アニメーションまたは演出テキストを出す
- participantToken を使って参加者画面で自分をハイライトする
- 役割配列を保存できるようにしておく（UIは簡易で可）

## Out of Scope
- 派手な抽選アニメーション
- ランキング演出の作り込み
- 再抽選機能
- 抽選結果編集
- QRコードUIの磨き込み
- Cronによる72時間削除
- WebSocket / Durable Objects
- E2Eテストの本格導入
- 管理者アカウント機能
- 複数ルーム横断の管理画面

## Detailed Design Notes
1. 抽選アルゴリズム
- 対象は締め切り時点の participants 一覧
- 参加者配列をシャッフルし、先頭から `winner_count` 件を採用する
- `ranked = true` のときのみ `rank: 1..n` を付与する
- `ranked = false` のときは rank を `null` または省略する
- roles は存在する場合、先頭から順に winners に割り当てる
- roles 数が winner_count より少ない場合、余りの role は `null`
- roles 数が winner_count より多い場合、超過分は無視する

2. 状態遷移
- `waiting`: 参加受付中
- `closed`: 参加締め切り済み、抽選前
- `drawn`: 抽選済み
- `waiting` 以外では join 不可
- `drawn` では再抽選不可
- draw API は `closed` でのみ受け付ける

3. 設定保存方針
- 抽選設定は `rooms.lottery_settings` に保存
- cycle 2 で最低限使うキー
  - `ranked`
  - `winner_count`
  - `roles`
- `show_result_to_participants` は別カラムを正として扱う

4. 結果取得方針
- 主催者向け: hostToken で常に取得可能
- 参加者向け: `show_result_to_participants = 1` のときのみ取得可能
- 主催者画面と参加者画面で見せ方は分けても、保存される結果データは1つにする

5. UI/UX方針
- cycle 2 は「動くこと」優先
- 抽選演出は最小でよく、数秒の疑似ローディングや順次表示で十分
- 設定入力は主催者トップ画面または draw画面のどちらか1箇所に寄せる

## Definition of Done
- 主催者が参加締め切りできる
- 締め切り後に新規参加が拒否される
- 主催者が抽選設定を保存できる
- 主催者が締め切り後に抽選を開始できる
- 抽選結果がDBに保存される
- 主催者画面に結果が表示される
- 公開ONのとき参加者画面に結果が表示される
- 公開OFFのとき参加者画面に非公開メッセージが表示される
- 同一ルームで抽選を2回実行できない
- 少なくとも `build` が成功する
